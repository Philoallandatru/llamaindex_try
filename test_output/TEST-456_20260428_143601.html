Root Cause Analysis

Based on the provided issue details and documentation, here is the Root Cause Analysis, Action Items, and Verification Steps for TEST-456.

1. Root Cause Analysis

The failure of the PCIe Gen4 NVMe SSD to enumerate on a cold boot (while functioning correctly after a warm reboot) indicates a power sequencing or initialization timing mismatch between the host controller (ASUS ROG motherboard) and the device firmware, specifically regarding the PCIe Base 5.0 specification requirements for link training.

Symptom Correlation: The fact that the device works immediately after a warm reboot suggests that the hardware is physically capable of communication, but the initial state upon power-on (cold boot) violates the deterministic timing rules required by the PCIe spec to successfully complete Link Training. Spec Violation: According to the `PCI_Express_Base_5.0r1.0` documentation provided: "Following a Conventional Reset, some devices may require additional time before they are able to respond to Requests... Particularly for Configuration Requests it is necessary that components and devices behave in a deterministic way." Hypothesis: The firmware initialization code (likely the `_DSM` or ACPI handler referenced in the `PCI_Firmware_v3.3` doc) does not include sufficient delay before issuing PCIe configuration requests during the cold boot sequence. Unlike a warm reboot, where the system state is preserved and the link is already established, the cold boot triggers a full reset cycle. If the firmware attempts to access the device's Configuration Space or trigger Link Training too quickly after power assertion (before the device stabilizes), it results in a timeout, causing `lspci` to fail to detect the device.

3. Verification Steps

To confirm the fix and ensure stability across different reboot scenarios:

1. Cold Boot Test: Power off the system completely (remove power or shut down OS). Perform a cold boot (power cycle). Immediately run `lspci -vvv` within 30 seconds of boot. Expected Result: The NVMe SSD appears in the list without requiring a subsequent reboot. No "Link Training Timeout" errors should appear in `dmesg`.

2. Warm Boot Test (Regression Check): Boot the system normally. Perform a standard OS shutdown and immediate power-on (warm boot). Run `lspci -vvv` and check `dmesg`. Expected Result: Device enumerates successfully with no latency increase or errors.

3. Hot-Plug Test: With the system running, unplug and replug the NVMe SSD (if supported by the chassis/controller). Monitor `dmesg` for AER (Advanced Error Reporting) messages or DPC event handling failures. Verify that the device is recognized immediately without needing a full system reboot.

4. Performance Validation: Run a benchmark (e.g., `fio`) to ensure sequential read/write speeds match baseline expectations (approx. 3500 MB/s for Gen4 NVMe). Confirm that the delay introduced in the firmware does not impact I/O latency during active usage.

5. Documentation Review: Verify that the updated flow chart in `PCI_Firmware_v3.3` correctly reflects the new "Wait for Deterministic State" step before granting control to the OS or initiating Link Training.
